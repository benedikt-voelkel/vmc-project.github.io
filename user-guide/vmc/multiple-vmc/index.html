Multiple VMC Engines

Overview

The VMC package allows splitting the event simulation among multiple engines. The criteria how to split the simulation have to be defined by the user and among others the decision which engine to be used could depend on

  • geometry
  • particle phase space
  • particle type
  • any combination of that
  • ...
Running multiple engines is fully supported for
  • GEANT3
  • GEANT4
This also allows the user to implement his/her own class deriving from TVirtualMC to be used in a splitted simulation. The communication between the engines is handled by a singleton object of type TMCManager such that the user does not have to deal with special implementation details. Hence, pausing an engines, re-starting them or transferring a track from one to the other is done automatically and at the same time the user stack is kept up-to-date. Furthermore, the same user application derived from TVirtualMCApplication together with the same user stack derived from TVirtualMCStack can be used for both a single engine run or a simulation splitted among multiple ones.

Further general information on the VMC project can be found here

Running multiple different engines

To keep the overhead as small as possible, the TMCManager object has to be requested by the user explicitly during construction of the UserApplication by calling TVirtualMCApplication::RequestManager(). That make the manager available via the protected member TVirtualMCApplication::fMCManager but a pointer can also always be obtained via TMCManager *TMCManager::Instance().

The most important interfaces of the TMCManager to deal with multiple VMCs are:

  • void SetUserStack(TVirtualMCStack* userStack)
    This notifies the manager about the user stack such that it will be kept up-to-date during the simulation. If that is not done, the TMCManager will abort.
  • void ForwardTrack(Int_t toBeDone, Int_t trackId, Int_t parentId, TParticle* userParticle)
    The user is always the owner of all track objects (aka TParticle) being created. Hence, all engine calls to TVirtualMCStack::PushTrack(...) are first redirected to the user stack where the user needs to make sure a TParticle object is created. After that, TMCManager::ForwardTrack(..) must be invoked to forward the pointer of the TParticle object which was just constructed. If a particle should be pushed to an engine other than the one currently running, the engine’s ID has to be provided as the last argument.
  • void TransferTrack(Int_t targetEngineId)
    For instance during TVirtualMCApplication::Stepping() the user might decide that the current track should be transferred to another engine, e.g., if a certain volume is entered. By specifying the ID or the pointer of the target engine the manager will take care of interrupting the track in the current engine, extracting the kinematics and geometry state and it will save these information for the target engine.
  • template <typename F> void Apply(F f)
    This assumes f to implement the () operator and taking a TVirtualMC pointer as an arument. f will then be called for all registered engines.
  • template <typename F> void Init(F f)
    This works similar to TMCManager::Apply during the initialization of the engines. It can also be called without an argument such that no additional user routine is included.
  • void Run(Int_t nEvents)
    A run involving all registered VMCs is steered for the specified number of events.
  • void ConnectEnginePointers(TVirtualMC *&mc)
    This gives the possibility for a user to pass a pointer of type TVirtualMC which will always be set to point to the currently running engine. Note, however, that there is a protected member TVirtualMCApplication::fMC which is always up-to-date and can hence be used within the user’s application code.
  • TVirtualMC *GetCurrentEngine()
    This provides the user with a pointer to the currently running engine.

An example of how the TMCManager is utilized in a multi-run can be found in the GEANT4_VMC repository. Looking at the diff between e.g. E03a and E03c highlights the very little additions necessary for running multiple engines. As one can see, minimal extensions are required and the same user stack and application can be used for both a single and a multi-run. In addition, this example can be used enabling only one engine or both GEANT3_VMC and GEANT4_VMC.

Example workflow

Implementation

  1. Implement your application as you have done before. Request the TMCManager in your constructor by calling TVirtualMCApplication::RequestMCManager().
  2. Implement your user stack as you have done before. At an appropriate stage (i.e. in UserStack::PushTrack(...)) you would call TMCManager::ForwardTrack(...) to forward the pointers to your newly constructed TParticle objects.
  3. Set your stack using TMCManager::SetUserStack(...).
To use the same stack in all cases, one can always check whether the manager is present checking if TMCManager::Instance() returns a valid pointer.

Usage/running

  1. Instantiate your application,
  2. instantiate the engines you want to use (these are registered automatically to the manager),
  3. call TMCManager::Init(...) (if needed with your custom initialization procedure),
  4. call TMCManager::Run(...) specifiyng the desired number of events to be simulated.

Further comments
The geometry is built once centrally via the TMCManager calling

  1. TVirtualMCApplication::ConstructGeometry(),
  2. TVirtualMCApplication::MisalignGeometry(),
  3. TVirtualMCApplication::ConstructOpGeometry().

so it is expected that these methods do not depend on any engine.

If multiple engines have been instantiated, never use TVirtualMC::ProcessRun(...) or other steering methods on the individual engines since that would bypass the TMCManager and will lead to inconsistent behavior.
